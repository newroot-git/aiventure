# AIventure Audit

## 1. Code quality & efficiency
- **Severity: Medium** — `components/PlanView.tsx:1-1079`  
  The main plan surface is a 1k+ line component that mixes data normalization, network orchestration, optimistic state, and UI. That makes every change risky and blocks reuse (calendar, history, future native shell).  
  **Fix:** Split into composable units: server data mappers (slot builder, recurrence), client hooks for optimistic writes, and presentational children (hero, slot list, actions). Keep side effects behind dedicated hooks.

- **Severity: Medium** — `lib/db.ts:188-244`  
  `writeMeta` does a read–modify–write without any guard. Two concurrent writers (e.g. owner + AI refine) can clobber scaffold/recurrence and corrupt meta.  
  **Fix:** Move meta to a dedicated table with a UNIQUE constraint and use `upsert`/`update` with `eq` filters on deterministic keys (plan_id + title). Alternatively use Supabase RPC to perform the merge atomically.

## 2. Bugs
- **Severity: High (data leak)** — `lib/db.ts:746-805`  
  `getFriends()` returns every profile (names, interests) to any signed-in or guest user; `/api/friends` exposes it. There is no relationship filter, so a forged guest session can enumerate the whole user base.  
  **Fix:** Introduce a real friendship/crew membership join table and scope this query accordingly; block guests from accessing data they don't share a plan/group with.

- **Severity: High** — `app/(app)/profile/edit/page.tsx:8-73`  
  The edit profile screen is still wired to `CURRENT_USER` mock data and never calls `/api/me`. Users can’t see current values or persist edits.  
  **Fix:** Convert to a server component that loads `getCurrentProfile()` or call `/api/me` on mount. On save, POST to `/api/me` and handle optimistic UI/error states.

- **Severity: High** — `app/(app)/groups/new/page.tsx:9-75`  
  Group creation still hard-codes `MOCK_PEOPLE` and just `router.push("/groups")`. No group is persisted, so the core “crew” concept can’t be created.  
  **Fix:** Fetch real eligible members (`/api/friends`) and POST to a real `/api/groups/create` endpoint that persists the group + members.

- **Severity: High** — `app/(app)/nudge/page.tsx:13-87`  
  The “Send a nudge” flow is entirely mock-driven; it never calls `/api/nudge`, so sender and receiver never get a plan/inbox item.  
  **Fix:** Replace `MOCK_FRIENDS` with `/api/friends`, wire the CTA to POST `/api/nudge`, and display server errors/success state.

- **Severity: Medium** — `app/(app)/invites/page.tsx:8-66`  
  The invites inbox uses `MOCK_INVITES`; Accept only `router.push`es and Decline just mutates local state. Backend invites remain untouched, so users can’t clear real invites.  
  **Fix:** Load invites via `getInvites()` (server component or client fetch) and call `/api/nudge/respond`/`/api/plans/edit?action=rsvp` to accept/decline, then revalidate.

## 3. What could be done better
- **Severity: Medium** — `lib/db.ts:736-808`  
  “Friends” are inferred as “any other profile”; that makes features (group picker, nudge picker) unusable for real networks and forces leaking data.  
  **Fix:** Define an explicit relationship model (friendships or shared-plan history), update `getFriends()` to honour it, and update client pickers to show only legitimate connections.

- **Severity: Medium** — `app/new/page.tsx:122-146`  
  The planner rehydrates preferences from `localStorage`. Once multiple devices are involved the state diverges from Supabase profile data.  
  **Fix:** Read interests/home area from `/api/me` (server state) and treat `localStorage` as a cache only, syncing changes immediately via API.

## 4. Lag, speed, optimization
- **Severity: Medium** — `components/PlanMap.tsx:121-170`  
  Each pin geocode runs sequentially (`for await`), so a 6-stop itinerary does 6 serial network requests (~6× slower).  
  **Fix:** Build the lookup list first and `await Promise.all(...)`, reusing the existing cache map to stay within Nominatim limits.

- **Severity: Medium** — `components/PlaceSearch.tsx:33-96` & `components/PlanMap.tsx:87-170`  
  All Nominatim requests omit the required `User-Agent`. The service will throttle/403, leaving users with empty results/maps.  
  **Fix:** Pass a descriptive `User-Agent` (and optional email) header on every Nominatim request and handle 429s with UI feedback/retry.

## 5. UI improvement potential
- **Severity: Low** — `app/(app)/g/[id]/page.tsx:29-50`  
  The “Add” avatar button is a dead end—no sheet or tooltip—so crews can’t discover how to add members.  
  **Fix:** Hook it up to the same invite sheet used in `PlanView`, or at least open guidance text/sheet instead of a no-op button.

- **Severity: Low** — `app/(app)/new/page.tsx:182-214`  
  When geolocation fails (“Use my location”), the UI silently resets to idle; users don’t know why nothing happened.  
  **Fix:** Surface an inline error/toast when permission is denied or reverse geocode fails, and suggest typing an area manually.

## 6. Concept & product direction
- **Severity: High** — `app/(app)/groups/new/page.tsx:9-75`, `app/(app)/nudge/page.tsx:13-87`, `app/(app)/invites/page.tsx:8-66`, `app/(app)/profile/edit/page.tsx:8-73`  
  Core social loop pillars (forming crews, nudging friends, responding to invites, updating your profile) are still stubbed with mock data. That means the “AIventure” promise—activate your real crew—breaks immediately after login.  
  **Fix:** Prioritize replacing every mock-backed screen with the real Supabase workflows and harden the permission model; without that loop the product can’t prove the concept.
