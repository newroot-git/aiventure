## 1. Code quality & efficiency
- **Severity: Medium** — `lib/db.ts:1-1442`  
  Monolithic 1.4k-line module mixes auth, plan domain logic, notifications, demo seeding, and SQL string-building. It’s nearly impossible to reason about side effects or share logic safely.  
  **Fix:** Split into focused services (e.g., `auth.ts`, `plans.ts`, `nudges.ts`, `groups.ts`) and move common Supabase helpers + row mappers into shared utilities. Establish typed return DTOs so routes can import only what they need.

- **Severity: Medium** — `components/PlanView.tsx:1-1314`  
  Single client component handles every UI concern (RSVP, slot editing, map, recurrence, invites). This bloats re-renders and makes regression risk enormous.  
  **Fix:** Break into self-contained subcomponents/hooks (e.g., `PlanHero`, `SlotsSection`, `LogisticsSection`, `InviteSheet`). Co-locate optimistic state in dedicated hooks so the main tree just composes pieces.

- **Severity: Low** — `app/page.tsx:1-492`  
  Huge inline CSS string embedded in the component makes the landing page un-diffable and prevents reuse/theme overrides.  
  **Fix:** Extract to CSS/SCSS (with tokens) or to styled components so the marketing page can be iterated on without editing a 500-line template literal.

## 2. Bugs
- **Severity: High** — `components/PlanView.tsx:457`  
  `const [rsvp, setRsvpState] = React.useState<RSVP>(myRsvp ?? "in");` renders newcomers as already “I’m in” even when they’ve never responded. Users can bounce thinking they’re counted, but the backend never records them.  
  **Fix:** Default to `myRsvp ?? "maybe"` (or `null` + explicit CTA) and only show the green “in” state after `/api/plans/edit` succeeds.

- **Severity: High** — `lib/db.ts:214-229`  
  Trip generation tells Claude to plan a `${days}-day trip` where `days = input.nights || 2`. Two nights should yield three days, but the prompt (and resulting scaffold) miss the final day.  
  **Fix:** Derive days as `const days = Math.max(1, (input.nights ?? 1) + 1);` and keep the scaffold’s `nights + 1` in sync.

- **Severity: Medium** — `components/PlanMap.tsx:78-105`  
  Nominatim requests lack the required `User-Agent`/`referer`. Once the upstream rate-limiter tightens (common in production), location lookup will start returning 403 and the map never loads.  
  **Fix:** Add an identifying header (per Nominatim policy) via `fetch(url, { headers: { "User-Agent": "...", "Accept-Language": "en" } })` and consider proxying through your backend to centralize rate limiting.

## 3. What could be done better (architecture, data, API)
- **Severity: Medium** — `lib/db.ts`  
  Domain logic is tightly coupled to Supabase row shapes. There’s no DTO/schema validation before handing objects to the client, so any DB migration risks runtime crashes.  
  **Fix:** Introduce zod/Valibot schemas for DB responses and map them to stable app types before returning from service functions.

- **Severity: Medium** — `app/api/plans/*`  
  API handlers manually switch on `action` strings with no payload validation. A malformed client payload silently no-ops (`else return NextResponse.json({ error: "bad action params" }`) but still counts toward rate limits.  
  **Fix:** Define per-action schema guards (zod) and reject upfront with 400 so the service layer never sees garbage.

- **Severity: Low** — `components/PlanView.tsx`  
  Optimistic state for slots/recurrence is reimplemented inline. Without shared hooks, future mutations (photos, comments) will repeat this pattern and diverge.  
  **Fix:** Extract a `usePlanOptimisticState` hook that centralises reconciliation logic (status, chosen options, recurrence) and can be reused by other plan surfaces.

## 4. Lag, speed, optimization
- **Severity: Medium** — `lib/db.ts:940-1095` (`getPlanBySlug`)  
  Loads **all** `plan_options` rows (incl. meta payload) on every plan view, even though the client only needs activity options + dates. For large itineraries this is 10s of KB per request.  
  **Fix:** Split meta/date fetches: select `plan_options` filtered by `kind='activity'` and `title='date'`, fetch meta via a dedicated query, and omit unused columns (e.g., `image_url`, `suggested_by`).

- **Severity: Low** — `components/PlanMap.tsx:129-174`  
  Geocoding runs sequentially for each slot. Trips with many steps block on round-trips, delaying the map paint.  
  **Fix:** Collect unresolved pins and `await Promise.all` on the geocode promises before mutating state; throttle with a concurrency limiter if needed.

- **Severity: Low** — `app/(app)/plans/page.tsx:14-64`  
  `freshIdeas` recalculates on every request using `new Date()` causing identical users on the same day to get cache misses and prevents caching the home page.  
  **Fix:** Move idea selection into a deterministic helper keyed on `dayKey` and user interests, and memoize/cache it server-side so the page can leverage Next caching.

## 5. UI improvement potential
- **Severity: Medium** — `components/PlanView.tsx:472-520`  
  With the RSVP default bug, the call-to-action to join is invisible. Even after fixing state defaults, there’s no copy telling guests what happens when they tap.  
  **Fix:** Show a banner “You’re not counted yet — pick an RSVP” until the server confirms, and surface member count changes immediately.

- **Severity: Low** — `components/PlanView.tsx:986-1053`  
  When the owner picks an option, participants can no longer see the alternatives that were rejected. That’s confusing if someone wants to revisit an idea later.  
  **Fix:** Keep the collapsed stack but add a “View other options” accordion for non-owners so context isn’t lost.

- **Severity: Low** — `app/(app)/nudge/page.tsx:53-118`  
  Friends grid has no empty state CTA (e.g., “Invite someone to a plan to unlock nudges”).  
  **Fix:** Add a link to start a plan or share an invite right in the placeholder copy to guide new users.

## 6. Concept & product direction
- **Severity: High** — Product gap  
  Core differentiator (“AI Drop with grounded sources”) isn’t exposed to invitees: visitors landing on `/p/[slug]` can’t trigger a fresh drop or see why the options are trustworthy.  
  **Fix:** Surface a lightweight “Regenerate ideas” (guarded by owner) with the grounded chip and link to the source so the anti-hallucination story is visible to everyone on the plan.

- **Severity: Medium** — Product gap  
  No follow-up loop after completion: the Adventure card exists, but there’s no share/export button visible once the plan is marked done (only internal share).  
  **Fix:** Add explicit “Save / Share” actions (download image, copy link) and aggregate Adventure history per person to reinforce the “record of a life lived” narrative.

- **Severity: Medium** — Scope alignment  
  Communities/open events are stubbed out (Explore shows “coming soon”). Without at least one live open plan, the discovery pillar in INTENT.md is missing.  
  **Fix:** Either hide the entire section until real data exists or ship a minimal open-plan pipeline (e.g., seed 2 curated public plans per metro) so the differentiator is tangible.
