## 1. Code quality & efficiency
- **Severity:** Medium  
  **Location:** app/new/page.tsx:154-188 & components/PlaceSearch.tsx:29-74  
  **Issue:** All OpenStreetMap/Nominatim requests omit the mandatory `User-Agent` (and ideally contact email). OSM will rate-limit or reject anonymous traffic, so address search quietly fails in production.  
  **Fix:** Pass an identifying header on every Nominatim call, e.g. `headers: { "User-Agent": "AIventure/1.0 (contact@aiventure.app)", "Accept-Language": "en" }`, and consider proxying via your own backend to respect usage rules.

- **Severity:** Medium  
  **Location:** app/(app)/nudge/page.tsx:13-78  
  **Issue:** Page still hard-wired to `MOCK_FRIENDS`, so production users can only “nudge” fake people. This is dead code that diverges from the real nudge API.  
  **Fix:** Replace mock state with the same `/api/friends` fetch used elsewhere (or server-side load), and POST to `/api/nudge` instead of toggling local state only.

- **Severity:** Medium  
  **Location:** app/(app)/invites/page.tsx:9-63  
  **Issue:** Invites UI is backed by `MOCK_INVITES`; accepting/declining never touches the real `/api/invites` data so the inbox can’t clear actual invites.  
  **Fix:** Load invites server-side via `getInvites()` (like the shell does) or hit `/api/notifications`, and wire the action buttons to the existing invite APIs.

## 2. Bugs
- **Severity:** Critical  
  **Location:** app/(app)/layout.tsx:7-16 & lib/db.ts:313-322 (`getAllProfiles`)  
  **Issue:** `getAllProfiles()` runs for every user, and the result (including names/emails/notes) is serialized into the client payload even when `NEXT_PUBLIC_DEV_SWITCH` is off. Any logged-in guest can pull every profile’s PII from the page JSON.  
  **Fix:** Gate the query server-side: only fetch when `process.env.NEXT_PUBLIC_DEV_SWITCH === "1"` *and* the requester is allowed, or move the dev switcher behind a server action that responds 404 in prod.

- **Severity:** High  
  **Location:** lib/db.ts:364-395 (`getGroup`)  
  **Issue:** No membership/ownership check before returning group details and plan list. Any authenticated user (or guest) who guesses a UUID can enumerate crew rosters and private plan slugs.  
  **Fix:** Call `currentUserId()` and verify the user is the group owner/member (via `group_members`) before returning; otherwise throw `not your group`.

- **Severity:** High  
  **Location:** lib/db.ts:744-781 (`invitePeople`)  
  **Issue:** New invitees are inserted into `plan_members` with `rsvp: "in"`, so they appear as going (and are counted in lock logic) before they’ve accepted. That breaks availability counts and “poke the undecided” logic.  
  **Fix:** Upsert invited members as `rsvp: "maybe"` (or null) until they explicitly accept; update the RSVP in `respondNudge`/accept flows.

## 3. What could be done better
- **Severity:** Medium  
  **Location:** lib/db.ts:1-1217  
  **Issue:** One mega-module owns identity, authorization, plan CRUD, notifications, nudges, recurrence, etc. It’s untestable and any change risks regressions.  
  **Fix:** Break it into cohesive modules (auth helpers, plan reads, plan writes, notifications, nudges) and export smaller surfaces; keep shared helpers (e.g., `planAuth`) in a dedicated file.

- **Severity:** Medium  
  **Location:** lib/db.ts:140-176 (`writeMeta`)  
  **Issue:** Meta writes are read-modify-write updates without row-level locking or unique constraints. Concurrent edits (e.g., slot add + recurrence change) can silently clobber each other.  
  **Fix:** Add a unique constraint on `(plan_id, title)` in `plan_options`, and update via SQL atomic statements (`insert ... on conflict do update`) so two writers can’t drop fields.

- **Severity:** Low  
  **Location:** components/PlanView.tsx:1-1038  
  **Issue:** PlanView mixes rendering, state, network orchestration, and helper components in a 1k-line client file. Maintenance and testing are painful, and memoization relies on manual discipline.  
  **Fix:** Extract slot controls, invite sheet, recurrence editor, and general feedback into dedicated components/hooks with their own files and tests.

## 4. Lag, speed, optimization
- **Severity:** High  
  **Location:** app/(app)/layout.tsx:7-16  
  **Issue:** Layout always materializes invites, nudges, notifications, profile, **and all profiles** for every page render, even when the user never opens the inbox/dev switcher. That’s several round trips on every navigation.  
  **Fix:** Load heavy inbox data lazily (e.g., via SWR in `NotificationsMenu`) and drop the global `getAllProfiles` fetch unless the dev switcher is actually visible.

- **Severity:** Medium  
  **Location:** components/PlanMap.tsx:23-147  
  **Issue:** Leaflet JS/CSS are pulled from CDN on every plan view; there’s no module-level caching beyond a Promise, and the tiles always reinitialize, adding ~100kB and layout jank per visit.  
  **Fix:** Gate the map behind an `IntersectionObserver`/dynamic import, or render a static map preview and let the user expand into Leaflet to avoid loading it for every visit.

- **Severity:** Low  
  **Location:** app/new/page.tsx:140-212  
  **Issue:** Location search hits Nominatim on every keystroke (after 400 ms) with no client cache, so typing “San Francisco” fans out 8 identical requests.  
  **Fix:** Cache recent query → result sets in component state (or SWR) keyed by query+area; reuse results instead of re-fetching.
