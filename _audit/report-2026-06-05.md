### 1. Code quality & efficiency
- **Severity: Medium** — `components/PlanView.tsx:1-1123` — Single 1k+ line component is mixing optimistic data orchestration, Supabase mutations, slot-building logic, and rendering. This makes regression risk huge and blocks reuse/testing.  
  **Fix:** Extract a `usePlanState` hook for optimistic mirrors and persistence helpers, split slot rendering into separate files, and keep `PlanView` as a thin presenter.

- **Severity: Medium** — `app/(app)/new/page.tsx:186-238` — Each keystroke hits Nominatim directly from the client with duplicated throttling logic (also repeated in `PlaceSearch.tsx`). There’s no shared caching, no required `User-Agent`, and no circuit breaker, so we risk 403s/rate-limit bans and waste bandwidth.  
  **Fix:** Move geocoding behind a server API/util that adds caching + UA header, reuse it in both onboarding and plan creation, and debounce centrally.

### 2. Bugs
- **Severity: High** — `components/PlanView.tsx:247-267` — The “reconcile per-field” block calls `setOptStatus/setOptPlan/setOptChosen` during render when server data catches up. React 19 treats state updates during render as an error, so once the optimistic field matches the server value you hit “Too many re-renders”.  
  **Fix:** Move these resets into `useEffect` watchers on `plan`, `options`, and `recurrence` so reconciliation happens post-render.

- **Severity: High** — `app/(app)/invites/page.tsx:9-60` — Page still uses `MOCK_INVITES`. Real invites loaded via Supabase never render, “Accept” just pushes to the plan without hitting `/api/plans/edit`, and Decline doesn’t touch the backend.  
  **Fix:** Convert to a server component that calls `getInvites()` and wire the buttons to the real invite/RSVP endpoints.

- **Severity: High** — `lib/db.ts:678-707` — `getFriends()` selects every profile for every logged-in user (service-role bypassing RLS), then exposes names/interests for people you’ve never connected with. Any authenticated account can enumerate the user base.  
  **Fix:** Introduce an explicit friendship / shared-plan join table, scope queries to those IDs, and select only the minimal columns needed for the picker.

- **Severity: Medium** — `app/api/me/route.ts:24-34` — Onboarding sends `home_area` but the POST schema discards it, so `updateMyProfile` never sees the field. Everyone defaults to London despite choosing another city.  
  **Fix:** Extend the body type to include `home_area`, validate length, and forward it to `updateMyProfile`.

- **Severity: Medium** — `lib/db.ts:133-191` — `writeMeta()` does a read-modify-write with no uniqueness/locking. Two concurrent slot edits can both insert a `__meta` row, leaving duplicate scaffolds and inconsistent recurrence state.  
  **Fix:** Enforce a unique constraint on `(plan_id,title)` and change the write to `upsert` (or wrap in a SQL function/transaction) so only one canonical meta row exists.

### 3. What could be done better
- **Severity: High** — `lib/db.ts:38-711` — Every API route runs through the service-role client, so even with RLS enabled you’re trusting hand-written guards. One missed check (like `getFriends`) compromises all data.  
  **Fix:** Move read paths to the RLS-protected server client (`supabaseServer`) and reserve service-role for the few admin mutations that need it, with explicit stored procedures where necessary.

- **Severity: Medium** — `app/(app)/groups/new/page.tsx:9-74` — “Create group” is a dead stub built on `MOCK_PEOPLE`. Users hit “Create” and nothing persists, which breaks the core “crew” concept.  
  **Fix:** Replace mock data with real `getFriends()` output, call a new `/api/groups` POST that inserts the group + members, and show success/error feedback.

- **Severity: Medium** — `lib/db.ts:678-707` — The “friends” story is currently “all users”. Product flows (nudges, invites) will remain noisy until you model mutual consent (friend requests, shared groups, or inferred contacts).  
  **Fix:** Design a friend/crew graph (e.g., `friendships` table), require opt-in before surfacing someone in pickers, and cache the result client-side.

### 4. Lag, speed, optimization
- **Severity: Medium** — `lib/db.ts:678-707` — Even aside from privacy, `getFriends()` does a full-table scan and loads every profile blob, so the invite sheet and plan builder degrade linearly with user count.  
  **Fix:** Limit the query to the caller’s friend IDs, select only `id,name,avatar`, and paginate if the list can still be long.

- **Severity: Low** — `components/PlanMap.tsx:123-157` — Geocoding runs sequentially (`for await` with `await` inside the loop). A four-stop plan waits on each network call in series, so maps feel slow.  
  **Fix:** Batch geocode via `Promise.all` with a small concurrency limit, and reuse cached results before firing new requests.

### 5. UI improvement potential
- **Severity: Medium** — `app/onboard/page.tsx:43-92` — “Use my location” silently fails (errors are swallowed) and the field remains blank, so users don’t know they should type a place manually.  
  **Fix:** Show a toast/error state when geolocation or reverse geocode fails and surface the fallback instructions inline.

- **Severity: Low** — `app/(app)/calendar/page.tsx` & `components/CalendarView.tsx:76-140` — Multiple plans on the same day render identical tiles with only one time, so it’s impossible to tell which is which before clicking.  
  **Fix:** Display stacked badges or a popover listing all plans for the day (with titles and times) instead of reusing the first card’s metadata.

### 6. Concept & product direction
- **Severity: High** — Core social loop is incomplete: there’s no working group creation, no friend graph, and “invites” are mock data. That blocks the stated promise (“AI powers your existing crew”).  
  **Fix:** Prioritise real group CRUD, a minimal friendship model, and live invite acceptance so the planning loop works end-to-end.

- **Severity: Medium** — No-install participation is only half-implemented: links land on authenticated views that assume Supabase sessions, and there’s no email/calendar handoff for guests.  
  **Fix:** Add a public plan view that respects plan visibility, lets guests RSVP with email, and triggers calendar delivery without forcing sign-up.

- **Severity: Medium** — Adventure history is just a list; there’s no shareable “adventure card” export or per-person log integration, which weakens the “trophy” differentiator.  
  **Fix:** Add a “Save/Share card” flow that renders a downloadable image and surface completed adventures on the profile with filters and stats.
